# Задание на проектную (учебную) практику

## Введение

Задание на проектную (учебную) практику разработано для студентов первого курса, обучающихся по направлениям подготовки, связанным с информационными технологиями и информационной безопасностью. Трудоёмкость практики составляет **72 академических часа**. Задание может выполняться **индивидуально** или **в составе группы до 3 человек**.

Для управления версиями используется **Git**, для написания документации — **Markdown**, для создания статического веб-сайта — **HTML и CSS** (допускаются любые инструменты: ручная вёрстка, генераторы статических сайтов, применение LLM и пр.). В качестве платформы для размещения репозитория рекомендуется использовать **GitVerse** (https://gitverse.ru/). Допускается использование любых других систем, согласованных с руководителем практики.

**Общая структура работы:**

- **Обязательная часть** – осваиваются базовые навыки Git, Markdown, создаётся сайт по проекту по дисциплине «Проектная деятельность», изучаются основы LLM, организуется взаимодействие с организацией-партнёром, оформляется итоговый отчёт. Трудоёмкость – около 40 часов.
- **Часть по выбору** (тоже обязательная, выполняется один из предложенных ниже вариантов) – углублённая работа: кафедральное задание, создание технологии «с нуля» или вклад в открытый проект. Трудоёмкость – около 32 часов.

## 1. Обязательная часть

### 1.1. Настройка Git и репозитория

- Зарегистрируйтесь на платформе **GitVerse**.
- Перейдите по предоставленной ссылке на GitVerse и **примите приглашение в «задание»**. Как только примете задание, откроется доступ к репозиторию, в котором нужно будет продолжить работу.
- Настройте локальную среду:
  - установите Git на свой компьютер;
  - настройте имя пользователя и email в Git:
    ```bash
    git config --global user.name "Ваше Имя"
    git config --global user.email "your.email@example.com"
    ```
- Свяжите локальный Git с аккаунтом GitVerse (через SSH‑ключ).
- Изучите [документацию по GitVerse](https://gitverse.ru/docs/gitverse/) и [онлайн-книгу по Git](https://git-scm.com/book/ru/v2).
- Освойте базовые команды Git: клонирование, коммит, создание веток, слияние, отправка изменений (push).
- Склонируйте ваш новый личный репозиторий задания на свой компьютер (clone).
- Регулярно фиксируйте изменения с осмысленными сообщениями коммитов по ходу выполнения следующих задач.
- **Ожидаемая трудоёмкость:** 5 часов.

### 1.2. Изучение Markdown и создание документации проекта

- В GitVerse в своём репозитории создайте **задачу** (Issue) с названием «Изучение Markdown и создание документации проекта». В описании укажите задачи текущего параграфа. Назначьте задачу себе: укажите исполнителя. Установите контрольный срок.
- Изучите синтаксис Markdown (заголовки, списки, ссылки, изображения, таблицы, блоки кода).
- В локальном репозитории папке `docs/` создайте набор markdown-файлов, описывающих ваш проект по дисциплине «Проектная деятельность». Эти документы по содержанию должны соответствовать страницам сайта (см. п. 1.3). Минимальный набор:
  - `docs/project.md` – описание проекта (цели, задачи, актуальность).
  - `docs/members.md` – участники группы и их личный вклад.
  - `docs/journal.md` – журнал прогресса (не менее трёх записей с датами).
  - `docs/resources.md` – полезные ссылки (организация-партнёр, статьи, курсы).
- Все markdown-документы должны быть аккуратными, содержать заголовки, списки и при необходимости иллюстрации.
- Зафиксируйте изменения в Git и отправьте их на GitVerse. В тексте коммита укажите «Документация проекта #номер_задачи», где вместо «номер задачи» подставьте номер вашего Issue (#TSK...).
- Отправьте изменения в GitVerse (push).
- Переведите созданную задачу (Issue) в статус «Готово».
- **Ожидаемая трудоёмкость:** 5 часов.

### 1.3. Создание веб-сайта

Создайте веб-сайт, посвящённый **основному проекту по дисциплине «Проектная деятельность»**, в котором вы участвуете.

**Инструменты:** любые – ручная вёрстка (HTML/CSS), генераторы статических сайтов (например, Hugo), помощь LLM. Главное – уникальность (не более 50% совпадения с работами других студентов) и соответствие требованиям.

**Обязательные страницы и разделы:**

1. **Главная страница** – аннотация проекта, ключевая информация.
2. **«О проекте»** – развёрнутое описание целей, задач, актуальности.
3. **«Участники»** – список участников группы с указанием личного вклада каждого в проект по «Проектной деятельности».
4. **«Журнал»** – минимум три поста (новости, заметки, отчёты о прогрессе) с датами и описанием выполненных шагов.
5. **«Ресурсы»** – полезные ссылки: организация-партнёр, использованные статьи, онлайн-курсы, документация и т.п.

**Оформление:**

- Добавьте графические материалы (фотографии, схемы, диаграммы, иллюстрации).
- По желанию используйте видео (например, вставку с YouTube или локальное видео).
- Сайт должен быть доступен для просмотра в браузере без дополнительных серверных настроек.

**Git:**

- В GitVerse в своём репозитории создайте **задачу** (Issue) с названием «Создание веб-сайта». В описании укажите задачи текущего параграфа. Назначьте задачу себе: укажите исполнителя. Установите контрольный срок.
- В локальном репозитории в папке site разработайте веб-сайт.
- Зафиксируйте файлы сайта в Git и отправьте их на GitVerse. В тексте коммита укажите «Создание веб-сайта #номер_задачи», где вместо «номер задачи» подставьте номер вашего Issue (#TSK...).
- Отправьте изменения в GitVerse (push).
- Переведите созданную задачу (Issue) в статус «Готово».

**Ожидаемая трудоёмкость:**

- Изучение HTML/CSS и вёрстка – 10–14 часов.
- Наполнение контентом и дизайн – 4–8 часов.

### 1.4. Онлайн-курсы по искусственному интеллекту (LLM)

В рамках практики необходимо ознакомиться с современными большими языковыми моделями.

**Задачи:**

- Пройти **один или несколько** онлайн-курсов (бесплатных или предоставленных вузом) по темам:
  - Введение в LLM (архитектура Transformer, принципы работы).
  - Промпт-инжиниринг (эффективное составление запросов)..
  - Этические аспекты использования ИИ.
  - Применение LLM в разработке и документировании.
- По завершении курсов подготовьте краткий отчёт в формате Markdown (2–3 страницы) с указанием:
  - названий курсов и ссылок на них;
  - полученных знаний;
  - примеров использования LLM при выполнении практики.
- Отчёт разместите в репозитории в папке `docs/` (например, `docs/llm_report.md`).

**Ожидаемая трудоёмкость:** 8–10 часов (в зависимости от объёма курса).

## 2. Часть по выбору (выполняется один из трёх вариантов)

Студент (или группа) выбирает **один** из трёх вариантов по согласованию с ответственным за практику.

### 2.1. Кафедральное индивидуальное задание

- Получите у руководителя практики индивидуальное задание, связанное с научной или прикладной деятельностью кафедры.
- Выполните задание, задокументируйте результаты.
- Интегрируйте отчёт и результаты (код, описание) в репозиторий (папка `src/` для кода, `docs/` для описания). Добавьте информацию на сайт.

### 2.2. Практическая реализация технологии «с нуля»

- Выберите любую технологию / программный компонент для реализации с нуля (например: интерпретатор, HTTP-сервер, базу данных, чат-бота на LLM, простейший компилятор, систему мониторинга и т.п.).
  *Тему необходимо согласовать с руководителем практики.*
- Проведите исследование: изучите принципы работы выбранной технологии, найдите теоретические материалы, разберите существующие аналоги.
- Реализуйте работающий прототип (минимально жизнеспособный продукт).
- Создайте подробное **техническое руководство (туториал)** в формате Markdown, включающее:
  - Пошаговые инструкции по созданию технологии с нуля.
  - Примеры кода с комментариями.
  - Иллюстрации (от 3 до 10): схемы архитектуры, UML-диаграммы, графики, скриншоты работы.
  - Раздел «Модификация» – опишите, какие улучшения или дополнительные функции вы добавили в проект сверх базовой реализации (творческая часть).
- Задокументируйте проект в репозитории, а также представьте его на созданном сайте (страница с описанием, ссылкой на туториал, видео).
- Подготовьте **финальный отчёт** (в свободной форме, но в хронологическом порядке опишите этапы работы; для группы – отдельно укажите индивидуальные планы участников).

### 2.3. Вклад в открытый проект

- Найдите открытый проект в системе контроля версий (например, на GitVerse, GitHub или другой доступной платформе).
  *Рекомендуется выбирать проекты, имеющие не менее 50 звёзд (или аналогичный показатель популярности).*
- Изучите код проекта, выберите нереализованную функциональность или ошибку (баг).
- Внесите свой вклад: исправьте ошибку, добавьте новую функцию, улучшите документацию.
- Оформите pull request (или запрос на слияние) в соответствии с правилами проекта.
- Документируйте свой вклад в формате Markdown:
  - Ссылка на проект.
  - Описание проблемы / задачи.
  - Краткое руководство по внесённым изменениям.
  - Ссылка на принятый pull request (или переписку с мейнтейнерами).
- Разместите в репозитории документацию в `docs/`, код (если свой) – в `src/`, продублируйте информацию на сайте.

## 3. Структура Git-репозитория

Актуальная структура приведена в файле `git_structure.md`. Все markdown-документы, кроме `README.md`, размещаются в папке `docs/`. Бинарные отчёты – в `reports/`. Сайт – в `site/`. Исходный код – в `src/`. Задание и шаблоны – в `task/`.

---

## 4. Контрольные сроки

Сроки выполнения этапов практики устанавливаются руководителем практики и доводятся до студентов в файле `terms.md` (размещается в репозитории). Рекомендуется предусмотреть промежуточные проверки:
- настройка репозитория и Markdown-документация – 1-я неделя;
- создание сайта – 2-я неделя;
- прохождение курсов по LLM, отчёт о партнёре – 3-я неделя;
- вариативная часть – 4–5-я недели;
- итоговый отчёт и загрузка в LMS – 6-я неделя.
